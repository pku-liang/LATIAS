# 分析流程
## 构建基础：
```
    架构图，问题框架，连接关系等
```

## 合法性检查
```
    维度匹配，映射连接关系匹配，循环匹配等
```

## 性能分析

1. 分析每个节点所对应的张量，根据张量复用关系找到连接点，确定每一层层级所对应的张量list，区分input张量和output张量 -- TBD -- 此处仍需要优化，目前的划分张量并没有完全考虑连接关系带来的影响，实际实现的是一个虚拟的写回模式，目前的实现方式更多的是针对mapping树进行分析，但缺少了对于trans节点所对应的数据搬运量的直接表示（目前是通过找到对应节点是否有trans来找到连接关系并计算延迟）
    1. 需要给每一个trans节点也划分具体的tensor，不能把trans忽略
    2. DFS的遍历树
    3. 当访问到计算节点，找出对应计算所需要的输入张量和输出张量，记录输入张量到consumer，输出张量到producer
    4. 如果当前计算节点的输入张量不属于producer，则5，否则6
    5. 不属于producer，向上遍历树，给每一个tile节点添加input tensor
    6. 属于producer，找到producer中对应变量的输出节点，找到该输出节点的输出路径与当前计算节点输入路径的公共节点，具体为：1）添加当前节点到输出路径；2）查看当前节点的后续是否有trans，如果后续没有trans，返回1），如果有trans则找到trans同级别的下一个节点中的第一个tile节点的level（如果同级别下的下一个节点是trans节点，则重复下一个），返回1）
2. 循环分析：
    1. DFS遍历mapping树
    2. 依次遍历每一个节点中每一个循环维度
    3. 在初始循环维度时计算当前节点与上层节点的输入输出数据搬运量大小，并留存当前节点所存储的变量，用于循环变化时可能的数据复用
    4. 在循环中逐次递归计算每一个子节点的延迟，在叶子节点Op时停止
    5. 在当前节点的循环下，根据子节点的scope类型整合延迟 -- 此处缺少对inter-tile的类型的通用分析
    6. 返回上一级，继续循环
    7. 其他：延迟将被划分为三类：输入，计算，和输出延迟
3. 
4. 内存容量分析 -- 缺少对trans节点的容量分析
